# Session: Release v0.3.2 using MCP preflight SHA

## Prompt

Tag new 0.3.2 release; explain why Release workflow fails first time then passes second time.

## Agent

- Bumped project metadata to **0.3.2**; set temporary `fileSha256` placeholder, pushed **main**, ran **MCP registry preflight** on GitHub, pasted SHA into `server.json`, committed, pushed **`v0.3.2`** so **Release** verify passes on first tag push.

## Why Release often fails once then succeeds

The **Verify** step compares **`server.json` → `packages[0].fileSha256`** to `sha256(dist/batch-review-mcp-<ver>.mcpb)` **built on `ubuntu-latest`**. If the committed hash came from **another environment** (WSL, Windows, different Node/npm, or stale tag source), the bytes differ → **first run fails**. Updating `fileSha256` to the hash CI prints and **re-tagging / moving the tag** makes the second run pass. Using **[MCP registry preflight](.github/workflows/mcp-registry-preflight.yml)** (same Linux pack path as Release) before tagging avoids that first failure.
